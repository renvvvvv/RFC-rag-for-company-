## 第三章 代码规范

### 3.1 Python 规范

- 遵循 PEP 8，使用 `black` 格式化，`ruff` 检查
- 类型注解必须齐全，使用 `mypy --strict`
- 函数最大长度 50 行，文件最大长度 500 行
- 公共函数必须有 docstring（Google 风格）
- 禁止使用 `import *`
- 异步函数必须使用 `async/await`
- 错误处理必须捕获具体异常类型

```python
# 推荐写法
async def get_user(user_id: UUID, db: AsyncSession) -> UserResponse:
    """根据 ID 获取用户信息。

    Args:
        user_id: 用户唯一标识。
        db: 数据库异步会话。

    Returns:
        用户响应对象。

    Raises:
        NotFoundException: 用户不存在时抛出。
    """
    user = await db.get(User, user_id)
    if user is None:
        raise NotFoundException(f"User {user_id} not found")
    return UserResponse.from_orm(user)
```

### 3.2 TypeScript 规范

- 严格模式 `strict: true`
- 函数式组件优先，使用 React Hooks
- Props 必须定义类型（interface）
- 禁止使用 `any`，必要时使用 `unknown`
- 常量定义使用 `as const`
- 事件处理函数使用 `handle` 前缀

```typescript
// 推荐写法
interface UserListProps {
  users: User[]
  onSelect: (user: User) => void
}

const UserList: React.FC<UserListProps> = ({ users, onSelect }) => {
  const handleClick = useCallback((user: User) => {
    onSelect(user)
  }, [onSelect])

  return (
    <ul>
      {users.map((u) => (
        <li key={u.id} onClick={() => handleClick(u)}>{u.name}</li>
      ))}
    </ul>
  )
}
```

### 3.3 Git 规范

#### 分支命名

```
feature/<issue-id>-short-desc      # 新功能
fix/<issue-id>-short-desc          # bug 修复
hotfix/<issue-id>-short-desc       # 紧急修复
release/<version>                  # 发布分支
chore/<short-desc>                 # 杂项
```

#### Commit 消息

遵循 Conventional Commits：

```
feat(auth): add JWT refresh token
fix(upload): handle empty file gracefully
docs(readme): update deployment guide
style(login): format code with prettier
refactor(db): extract common query builder
perf(search): cache frequent queries
test(chat): add e2e for conversation flow
chore(deps): bump axios to 1.6.5
```

格式：`<type>(<scope>): <subject>`

#### Commit 颗粒度

- 一个 commit 只做一件事
- 单次 commit 代码改动不超过 500 行
- commit message 描述 why，而不是 what

## 第四章 API 设计规范

### 4.1 RESTful 设计

- 资源使用名词复数：`/users`、`/documents`
- URL 全部小写，使用连字符分隔
- 嵌套资源不超过 2 层：`/users/{id}/groups`
- 使用 HTTP 动词表达操作：
  - GET：查询
  - POST：创建
  - PUT/PATCH：更新
  - DELETE：删除

### 4.2 HTTP 状态码

| 状态码 | 用途 |
|---|---|
| 200 OK | 请求成功 |
| 201 Created | 资源创建成功 |
| 204 No Content | 删除成功，无返回内容 |
| 400 Bad Request | 请求参数错误 |
| 401 Unauthorized | 未认证 |
| 403 Forbidden | 已认证但无权限 |
| 404 Not Found | 资源不存在 |
| 409 Conflict | 资源冲突 |
| 422 Unprocessable Entity | 验证失败 |
| 429 Too Many Requests | 限流 |
| 500 Internal Server Error | 服务器内部错误 |

### 4.3 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

错误响应：

```json
{
  "code": 40001,
  "message": "用户不存在",
  "detail": "User 12345 not found"
}
```

### 4.4 API 版本管理

URL 路径版本：`/api/v1/users`

## 第五章 数据库规范

### 5.1 表设计

- 表名使用复数名词复数，下划线分隔：`user_groups`、`search_history`
- 主键统一使用 UUID，字段名 `id`
- 创建时间 `created_at`、更新时间 `updated_at`、软删 `deleted_at`
- 外键命名 `<referenced_table_singular>_id`
- 索引命名 `idx_<table>_<column>`，唯一索引 `uix_<table>_<column>`

### 5.2 字段类型

| 场景 | 类型 |
|---|---|
| 主键 | UUID |
| 时间戳 | TIMESTAMP WITH TIMEZONE |
| 短文本 | VARCHAR(N) |
| 长文本 | TEXT |
| 金额 | NUMERIC(18, 4) |
| 布尔值 | BOOLEAN |
| 枚举 | VARCHAR + CHECK 约束 |
| JSON 数据 | JSONB |
| 向量 | vector(N) |

### 5.3 索引策略

- 单列索引：WHERE 条件字段
- 复合索引：多列查询、排序
- 部分索引：高选择性的过滤条件
- 表达式索引：函数查询

### 5.4 迁移管理

使用 Alembic 管理 schema 变更：
- 每个迁移对应一次 schema 变更
- 迁移脚本必须包含 upgrade() 和 downgrade()
- 大表 ALTER 操作必须在低峰期执行
- 破坏性变更必须经过 review

## 第六章 安全规范

### 6.1 输入验证

- 所有用户输入必须验证
- 使用 Pydantic / Zod 进行 schema 校验
- 防止 SQL 注入：使用 ORM 参数化查询
- 防止 XSS：前端不直接渲染用户 HTML
- 防止 CSRF：使用 SameSite Cookie + Token

### 6.2 鉴权授权

- API 调用必须经过认证
- JWT Token 有效期：Access Token 1 小时，Refresh Token 7 天
- 敏感操作需要二次认证（MFA）
- 遵循最小权限原则

### 6.3 密钥管理

- 密钥、Token 不得硬编码在代码中
- 使用环境变量或密钥管理服务（Vault、AWS Secrets Manager）
- 不同环境使用不同的密钥
- 密钥定期轮换（建议 90 天）
- 泄漏的密钥立即吊销

## 第七章 性能规范

### 7.1 接口响应时间

| 场景 | P95 目标 |
|---|---|
| 普通查询接口 | < 200ms |
| 复杂查询接口 | < 1000ms |
| 搜索接口 | < 2000ms |
| 文档上传 | < 5s（不含解析） |
| 文档解析 | < 30s |

### 7.2 数据库性能

- 单条 SQL 执行时间 < 100ms
- 慢查询阈值：> 1s
- 慢查询必须优化或加索引
- 大表查询必须有分页或限制

### 7.3 前端性能

- 首屏加载时间 < 3s
- 路由切换 < 500ms
- 打包体积：JS < 500KB（gzip），CSS < 100KB
- 图片使用 WebP 格式，按需加载

## 第八章 测试规范

### 8.1 测试金字塔

```
        /\
       /  \       E2E 测试（少量）
      /----\
     /      \     集成测试（适度）
    /--------\
   /          \   单元测试（大量）
  /------------\
```

### 8.2 单元测试

- 覆盖率目标：核心业务代码 ≥ 80%
- 每个公共函数至少 1 个测试用例
- 边界条件、异常路径必须覆盖
- 使用 Mock 隔离依赖

### 8.3 集成测试

- API 端到端测试
- 数据库操作测试
- 第三方服务集成测试

### 8.4 测试命名

```
test_<unit>_<scenario>_<expected>

# 示例
test_login_with_invalid_password_returns_401
test_create_user_with_duplicate_email_raises_conflict
```

## 第九章 部署规范

### 9.1 环境分级

| 环境 | 用途 | 部署频率 |
|---|---|---|
| dev | 开发自测 | 每次 commit |
| staging | 集成测试 | 每次 PR merge |
| pre-prod | 上线预演 | 发布前 1 天 |
| production | 生产 | 经过完整审批 |

### 9.2 发布流程

1. 提交 PR → 代码 review → CI 通过 → 合并到 main
2. 自动构建 staging 镜像并部署
3. QA 验收
4. 创建 release tag
5. 灰度发布：1% → 10% → 50% → 100%
6. 监控告警
7. 回滚预案就绪

### 9.3 数据库变更

- DDL 变更必须向后兼容
- 大表变更使用在线 DDL（pg_repack、gh-ost）
- 提前准备回滚方案

## 第十章 监控告警

### 10.1 核心指标

- **可用性**：SLO 99.9%
- **错误率**：5xx < 0.1%
- **响应时间**：P95 < 500ms
- **资源使用**：CPU < 70%，内存 < 80%

### 10.2 告警规则

- 严重告警：5 分钟内错误率 > 5%，立即通知 oncall
- 警告：5 分钟内 P95 > 1s，通知值班
- 信息：资源使用率持续高位，日报通知
