### 问题 5：代码改了维度但运行中的数据库未同步

**现象**：7-01 修改了 `Vector(768)` → `Vector(1024)` 的代码，Worker 重建后仍报 `expected 768 dimensions, not 1024`，向量始终写不进去。

**原因**：代码层面的 `Vector(1024)` 只在 `CREATE TABLE` 时生效。**已存在的表不会自动变更**。`text_chunk_vectors` 表建表时已是 `vector(768)`，之后即使代码改了，数据库列定义不变。

**技术原理**：pgvector 的 `vector(N)` 在 PostgreSQL 存储引擎层面校验数据长度。INSERT 时逐个 float 对比：
- 列要求 768 个 float → 给了 1024 个 → 直接拒绝
- `ALTER COLUMN ... TYPE vector(1024)` 会触发全表重写（每行从 3076 字节扩到 4100 字节）

**修复**：
```sql
ALTER TABLE text_chunk_vectors ALTER COLUMN embedding TYPE vector(1024);
ALTER TABLE image_frame_vectors ALTER COLUMN embedding TYPE vector(1024);
```

| 文件 | 改动 |
|------|------|
| 数据库运行时 | `ALTER TABLE` 将两张向量表的 embedding 列从 `vector(768)`/`vector(512)` 扩为 `vector(1024)` |

**教训**：代码改 Schema 后，必须同步对运行中的数据库执行 DDL 变更，否则容器重建多少次都没用。

### 问题 7：测试数据重复导致指标下降

**现象**：chinese 分词配置后，指标大幅下降（recall@1: 0.720 → 0.000）。

**原因**：测试脚本每次运行都会上传文件，导致同一个文件被上传 6 次，产生 6 个 doc_id。测试脚本只期望最后一个 doc_id，但检索可能返回前面的 doc_id，导致匹配失败。

**修复**：清理重复文档，只保留每个文件最新的一个。
```sql
DELETE FROM chunks WHERE doc_id IN (
  SELECT d.id FROM documents d
  WHERE d.kb_id = 'a7103b51-a5c3-42a1-9af4-ebabdc3de10f'
  AND d.id NOT IN (
    SELECT DISTINCT ON (filename) id FROM documents
    WHERE kb_id = 'a7103b51-a5c3-42a1-9af4-ebabdc3de10f'
    ORDER BY filename, created_at DESC
  )
);

DELETE FROM documents
WHERE kb_id = 'a7103b51-a5c3-42a1-9af4-ebabdc3de10f'
AND id NOT IN (
  SELECT DISTINCT ON (filename) id FROM documents
  WHERE kb_id = 'a7103b51-a5c3-42a1-9af4-ebabdc3de10f'
  ORDER BY filename, created_at DESC
);
```

**结论**：chinese 分词配置是正确的，之前的指标下降完全是测试数据重复导致的。清理后指标完全恢复。

3. 违背「宁可放过不能错杀」的清洗原则

**现象**：18_告警参考.pdf 出现 154,779 字符的超大 chunk。

**根因**：两个误判 + 一个缺失：

| 问题 | 旧代码 | 新代码 |
|------|--------|--------|
| `is_table` 过宽 | `'[TABLE]' in chunk` | `startswith` + `endswith` 精确匹配 |
| `is_directory` 过宽 | `'.....' in chunk and count > 3` | ≥5行含且 >50% |
| 目录块不拆分 | 直接追加 | 超大目录也走句子切分 |
| 缺兜底逻辑 | 无 | 句切后 >2000字符非表格块按 `\n` 硬切 |

**效果**：18_告警 chunks: 516→773, max: 154,779→1,465
