> - Re-rank: `POST /v1/rerank`，输入 `{"query": "...", "documents": [...], "top_n": N}`，输出 `{"results": [{"index": 0, "relevance_score": 0.95}]}`

### 4.2 详细说明

#### 模块A：音频RAG Pipeline

```
待设计范围
═══════════════════════════════════════════════════════════════════

[音频文件] → [音频解码] → [音频分段] → [ASR转录] → [Speaker标识]
                                              │
                                              ▼
                                   ┌─────────────────────┐
                                   │  文本Chunk + 时间戳  │
                                   │  + Speaker标签       │
                                   └──────────┬──────────┘
                                              │
                                              ▼
                                   [Embedding] → [Milvus索引]
                                              │
                                              ▼
                                   [检索: 关键词+时间范围+Speaker]

═══════════════════════════════════════════════════════════════════
```

- **触发条件**：Phase 3启动 或 用户明确提出音频检索需求
- **设计输入**：ASR引擎选型结果、音频文件格式支持范围（.mp3/.wav/.m4a/.flac）
- **关键决策**：
  - ASR引擎：Whisper / 讯飞 / 阿里云 / 用户自研
  - 音频分段策略：固定时长 / VAD语音活动检测 / 语义断句
  - 与视频RAG的音频轨道复用策略

---

#### 模块B：视频音频轨道提取

- **触发条件**：视频Pipeline需要支持"语音内容检索" 或 音频RAG需要复用视频资源
- **技术路径**：
  1. ffmpeg提取音频流：`ffmpeg -i video.mp4 -vn -acodec copy audio.aac`
  2. 音频流送入音频RAG Pipeline复用处理逻辑
- **关键决策**：存储策略（是否独立存储音频文件 vs 运行时提取）

---

#### 模块C：Speaker Diarization（说话人分离）

- **触发条件**：会议录音RAG 或 多说话人访谈音频处理
- **候选方案**：
  | 方案 | 优点 | 缺点 |
  |------|------|------|
  | pyannote.audio | 精度高，社区活跃 | 计算资源消耗大 |
  | WhisperX | 与Whisper ASR集成好 | 说话人数量需预设 |
  | 阿里云智能语音 | 云端API，免运维 | 私有化需额外适配 |
- **输出格式**：`{"speaker": "A", "start": 12.5, "end": 18.3, "text": "..."}`

---

#### 模块D：OCR精确提取

- **触发条件**：扫描件PDF解析准确率不达标 或 图片内表格需要结构化提取
- **技术路径**：
  - 基础OCR：PaddleOCR（已集成）
  - 版面分析：PaddleStructure / Unstructured `partition_pdf` 的 `ocr=True` 模式
  - 表格提取：PP-Structure表格识别模型
- **验收指标**：扫描件PDF文字提取准确率≥95%；表格结构还原准确率≥85%

## 5. 风险与应对策略

### 5.1 风险矩阵

```
影响程度
  高 │  [R3]视频资源消耗    [R5]权限复杂度    [R6]向量DB扩展性
     │        ●                  ●                  ●
     │
  中 │  [R1]Embedding接入   [R4]多模态效果    [R8]跨模态融合
     │        ●                  ●                  ●
     │
  低 │  [R2]字段级权限性能  [R7]大文件上传
     │        ●                  ●
     └─────────────────────────────────────────────────────────
            低                  中                  高     发生概率
```

### 5.2 风险详述

#### R1 — Embedding服务接入风险

| 属性 | 内容 |
|------|------|
| **描述** | 用户自研Embedding服务接口不稳定、延迟过高或吞吐不足 |
| **概率** | 中 |
| **影响** | 中 |
| **应对策略** | 1. 定义严格的API契约（兼容OpenAI Embedding接口）<br>2. 实现Embedding调用熔断降级（失败时回退到本地备用模型或拒绝服务）<br>3. 批量请求+异步队列削峰<br>4. 提供Embedding服务健康检查与自动重试 |
| **责任人** | 后端架构师 |

#### R2 — 字段级权限性能风险

| 属性 | 内容 |
|------|------|
| **描述** | 字段级ABAC权限在Milvus expr中执行，复杂规则导致检索延迟激增 |
| **概率** | 低 |
| **影响** | 低 |
| **应对策略** | 1. 权限规则预编译缓存（Redis缓存用户→expr映射）<br>2. 字段级权限批量预计算，落入Chunk元数据标签<br>3. Milvus expr长度监控，超长规则触发告警并降级为文档级权限<br>4. 定期清理无效权限规则 |
| **责任人** | 权限服务负责人 |

#### R3 — 视频资源消耗风险 ⭐

| 属性 | 内容 |
|------|------|
| **描述** | 视频处理（关键帧提取、VLM推理、OCR）消耗大量GPU/CPU资源，导致Worker节点过载 |
| **概率** | 高 |
| **影响** | 高 |
| **应对策略** | 1. 视频处理独立Worker队列，资源隔离（独立K8s节点池或独立宿主机）<br>2. 关键帧提取策略分级（快速模式：固定间隔；精确模式：场景检测）<br>3. 视频处理限流：并发处理数上限，队列溢出时拒绝新任务<br>4. 视频预处理转码降级（高分辨率视频先降采样）<br>5. 视频处理结果缓存，避免重复计算 |
| **责任人** | 基础设施负责人 |

#### R4 — 多模态检索效果风险

| 属性 | 内容 |
|------|------|
| **描述** | 图片/视频/文档混合检索时，不同模态的向量空间不一致导致效果下降 |
| **概率** | 中 |
| **影响** | 中 |
| **应对策略** | 1. 模态内先检索、模态间RRF融合，避免直接跨模态向量比较<br>2. 引入查询模态路由（文本查文本、图片查图片、混合时多路并行）<br>3. 建立评测基准，量化各模态检索效果<br>4. 预留ColPali等统一多模态Embedding升级路径 |
| **责任人** | 算法负责人 |

#### R5 — 权限系统复杂度风险 ⭐

| 属性 | 内容 |
|------|------|
| **描述** | RBAC+ABAC+标签权限+字段级权限组合后，规则冲突或逻辑复杂难以维护 |
| **概率** | 高 |
| **影响** | 高 |
| **应对策略** | 1. 权限策略分层：RBAC粗粒度 → 标签中粒度 → ABAC细粒度，逐层过滤<br>2. 权限规则冲突检测机制（单元测试覆盖核心冲突场景）<br>3. 权限变更审计日志，支持回溯<br>4. 权限计算结果缓存（用户×文档权限矩阵缓存TTL）<br>5. 定期权限审查工具（检测冗余/无效规则） |
| **责任人** | 后端架构师 |

#### R6 — 向量数据库扩展性风险 ⭐

| 属性 | 内容 |
|------|------|
| **描述** | Milvus在十亿级向量、高并发写入+查询混合负载下性能衰减 |
| **概率** | 中 |
| **影响** | 高 |
| **应对策略** | 1. Collection按知识库分片，避免单Collection过大<br>2. 读写分离：写入走Kafka异步缓冲，查询走Milvus QueryNode集群<br>3. 索引类型策略：IVF_FLAT平衡召回与性能；HNSW用于高召回场景<br>4. 监控Milvus核心指标（QPS、latency、index build time），触发扩容<br>5. 预留多Milvus集群联邦查询能力 |
| **责任人** | 基础设施负责人 |

#### R7 — 大文件上传风险

| 属性 | 内容 |
|------|------|
| **描述** | 大文件（>1GB视频/批量PDF）上传导致内存溢出、超时或网络中断 |
| **概率** | 中 |
| **影响** | 低 |
| **应对策略** | 1. 分片上传（Multipart Upload），支持断点续传<br>2. 前端上传进度可视化 + 后端流式接收<br>3. 文件大小限制 + 类型白名单 + 病毒扫描前置<br>4. 大文件上传路由到独立Nginx/MinIO直连，绕过API网关<br>5. 上传任务异步化，上传完成通知触发后续Pipeline |
| **责任人** | 前端+后端开发 |

#### R8 — 跨模态融合效果风险

| 属性 | 内容 |
|------|------|
| **描述** | 文本、图片、视频检索结果融合时，RRF权重不合理导致整体效果差 |
| **概率** | 中 |
| **影响** | 中 |
| **应对策略** | 1. RRF参数（k值）通过评测数据集网格搜索确定<br>2. 支持按查询意图动态调整模态权重（文本问题提升文本模态权重）<br>3. 融合阶段保留各模态原始分数，便于调试和分析<br>4. 建立跨模态评测集，持续监控融合效果<br>5. 预留Learned Fusion（如LightGBM排序融合）升级路径 |
| **责任人** | 算法负责人 |

## 6. 术语表

| 术语 | 英文全称 | 说明 |
|------|---------|------|
| **RAG** | Retrieval-Augmented Generation | 检索增强生成，通过外部知识检索提升LLM回答准确性的技术架构 |
| **Chunk** | — | 文档分块，将长文档切分为适合向量化的文本片段 |
| **Embedding** | — | 嵌入向量，将文本/图片等多模态数据映射到高维语义空间的稠密向量表示 |
| **ANN** | Approximate Nearest Neighbor | 近似最近邻搜索，向量数据库中高效检索相似向量的算法（HNSW/IVF等） |
| **RRF** | Reciprocal Rank Fusion | 倒数秩融合，将多路检索结果按排名加权合并的融合算法 |
| **RBAC** | Role-Based Access Control | 基于角色的访问控制，通过角色分配权限的模型 |
| **ABAC** | Attribute-Based Access Control | 基于属性的访问控制，通过用户/资源/环境属性动态判定权限的模型 |
| **VLM** | Vision Language Model | 视觉语言模型，可同时理解图片和文本的多模态大模型 |
| **ASR** | Automatic Speech Recognition | 自动语音识别，将语音信号转换为文本的技术 |
| **OCR** | Optical Character Recognition | 光学字符识别，从图片/扫描件中提取文字的技术 |
| **Diarization** | Speaker Diarization | 说话人分离，识别音频中"谁在什么时候说话"的技术 |
| **BM25** | — | 经典概率检索模型，基于词频和文档长度的关键词排序算法 |
| **NDCG** | Normalized Discounted Cumulative Gain | 归一化折损累计增益，衡量检索结果排序质量的指标 |
| **MRR** | Mean Reciprocal Rank | 平均倒数排名，衡量检索首条结果准确率的指标 |
| **HNSW** | Hierarchical Navigable Small World | 分层可导航小世界图，一种高效的ANN索引算法 |
| **IVF** | Inverted File Index | 倒排文件索引，向量量化后的近似检索索引结构 |
| **ColPali** | — | 基于视觉语言模型的文档图片端到端向量化方法，保留版面信息 |
| **P99** | 99th Percentile | 第99百分位延迟，衡量系统尾部延迟的指标 |
| **SLA** | Service Level Agreement | 服务等级协议，承诺的系统可用性指标（如99.9%） |
