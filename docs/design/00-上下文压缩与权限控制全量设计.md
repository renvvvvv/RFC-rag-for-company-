## 1. 项目概述

### 1.1 系统定位

本系统是一套**企业级私有化多模态RAG知识库系统**，在提供强大检索生成能力的同时，具备**业界领先的多层级权限控制和上下文安全能力**。

### 1.2 核心特征

| 特征 | 描述 |
|------|------|
| **全模态覆盖** | 文档/PDF/Word/Excel/图片/视频(画面)/链接，音频RAG待设计 |
| **用户群权限** | 支持用户群(UserGroup)对资源的批量权限控制，群继承机制 |
| **五级权限穿透** | 文件类型 → 文档 → 字段(Excel列/Word段落) → 标签 → 关键词级别 |
| **关键词敏感控制** | 全局配置关键词级别(0-4)，内容级实时拦截 |
| **API安全网关** | 调用minimax-m3等外部API时的三级安全策略(L4本地/L3脱敏/L2直接) |
| **权限感知压缩** | 上下文压缩时保留权限边界，防止权限标记丢失 |
| **多轮对话隔离** | 历史上下文按权限级别过滤，防止跨轮次泄露 |

### 1.3 设计原则

```
P1: 纵深防御    - 单点失效不会导致整体安全崩溃
P2: 最小权限    - 用户只能访问完成工作所需的最小信息
P3: 权限前置    - 在检索阶段就剔除无权限内容，而非检索后过滤
P4: 标记内联    - 权限信息嵌入内容，压缩/截断时不丢失
P5: 边界不跨越  - 不同权限级别的内容绝不混合压缩
P6: 全程审计    - 每次权限判定和API调用都有完整审计日志
```

## 3. 上下文压缩全量设计

### 3.1 四大风险

| 风险 | 场景 | 后果 |
|------|------|------|
| R1: 压缩导致权限标记丢失 | LLMLingua压缩后，[公开][内部][机密]Chunk混合成一段文本 | LLM无法区分机密信息 |
| R2: 多轮对话泄露 | Round 1检索到机密Chunk，Round 2权限降级但上下文仍在 | 低权限用户看到机密内容 |
| R3: 截断丢失标记 | Token超限时截断，权限标记被截掉 | 无标记内容无法判断权限 |
| R4: 重排序打散分组 | Re-rank后[公开]和[机密]Chunk相邻，被一起压缩 | 权限边界被打破 |

### 3.2 四大设计原则

```
P1: 权限标记内联不可分
    权限信息必须嵌入文本内容中，使用XML格式
    <perm level='L3' doc='doc_003'>机密内容</perm>
    → 即使被压缩/截断，标记仍然保留

P2: 权限边界不跨越
    压缩操作只能在相同权限级别内进行
    [公开组]独立压缩 ←→ [内部组]独立压缩 ←→ [机密组]独立压缩
    → 不同级别绝不混合

P3: 历史上下文重新过滤
    每轮对话都重新检查历史消息的可见性
    用户权限变化 → 历史上下文动态过滤
    → Round N只能看到权限<=当前权限的历史

P4: 截断策略权限感知
    Token超限时，优先截断低权限内容
    保留高权限内容（因为用户有权限看，更"贵"）
    → 截断点必须在</perm>标记之后
```

### 3.3 权限分块压缩

```python
class PermissionAwareCompressor:
    """权限感知压缩器"""
    
    async def compress(self, chunks: List[Chunk], user: User) -> List[CompressedBlock]:
        # Step 1: 按权限级别分组
        groups = self._group_by_permission_level(chunks)
        # { "L0": [chunk_A, chunk_B], "L2": [chunk_C], "L3": [chunk_D] }
        
        # Step 2: 组内压缩（同权限级别可安全压缩）
        blocks = []
        for level, group_chunks in groups.items():
            combined = self._build_group_text(group_chunks, level)
            compressed = await llm_lingua.compress(combined)
            blocks.append(CompressedBlock(
                text=compressed,
                level=level,
                sources=[c.source for c in group_chunks]
            ))
        
        # Step 3: 权限感知截断（优先截断低权限）
        return self._permission_aware_truncate(blocks, max_tokens)
    
    def _build_group_text(self, chunks, level):
        """构建带内联标记的组文本"""
        parts = []
        for i, chunk in enumerate(chunks, 1):
            header = f"<source id='{i}' doc='{chunk.doc_id}' level='{level}'>"
            footer = "</source>"
            parts.append(f"{header}{chunk.text}{footer}")
        return "\n\n".join(parts)
```

### 3.4 多轮对话权限隔离

```python
class PermissionAwareConversation:
    """权限感知的对话管理"""
    
    def get_visible_history(self, current_user: User) -> List[Message]:
        """
        获取当前用户可见的历史消息
        每轮调用都重新过滤！
        """
        visible = []
        for msg in self.history:
            # 检查用户权限级别是否 >= 消息级别
            if self._user_has_level(current_user, msg.permission_level):
                visible.append(msg)
        return visible

# 示例:
# 历史: [Round1:L3消息] [Round2:L1消息] [Round3:L4消息]
# 用户权限=L2:
#   → 可见: [Round2:L1消息]
#   → 过滤: [Round1:L3] [Round3:L4]（权限不够）
```

### 3.5 安全截断

```python
def safe_truncate(context: str, max_tokens: int) -> str:
    """
    安全截断: 确保不会在权限标记中间截断
    """
    if estimate_tokens(context) <= max_tokens:
        return context
    
    # 找到所有</perm>的位置
    end_positions = []
    idx = 0
    while True:
        pos = context.find("</source>", idx)
        if pos == -1:
            break
        end_positions.append(pos + len("</source>"))
        idx = pos + 1
    
    # 从后向前找最后一个完整</source>之后截断
    for end_pos in reversed(end_positions):
        truncated = context[:end_pos]
        if estimate_tokens(truncated) <= max_tokens:
            return truncated
    
    # 最坏情况: 返回第一个完整块
    first_end = context.find("</source>")
    if first_end != -1:
        return context[:first_end + len("</source>")]
    
    return "<system>上下文过长，请简化查询</system>"
```

## 6. 安全模型

### 6.1 威胁模型

| 威胁 | 风险等级 | 防御措施 |
|------|----------|----------|
| 用户越权访问 | 高 | 五级权限穿透 + 关键词拦截 |
| 群权限继承错误 | 高 | DENY优先 + 显式权限覆盖 |
| 压缩导致信息泄露 | 高 | 权限分块压缩 + 标记内联 |
| 多轮对话泄露 | 中 | 每轮重新过滤历史 |
| API调用数据泄露 | 高 | 三级安全策略 + 脱敏 + 审计 |
| LLM编造敏感信息 | 中 | 返回结果校验 + 编造检测 |
| 缓存污染 | 中 | ACL版本号 + 增量更新 |
| 关键词绕过 | 中 | 变体匹配 + 语义匹配 |

### 6.2 安全假设

```
安全假设:
1. Redis缓存是可信的（部署在内网，有访问控制）
2. PostgreSQL是可信的（有备份和主从同步）
3. minimax-m3 API提供商不会泄露数据（通过L3脱敏降低风险）
4. 用户不会共享JWT Token（通过短TTL和刷新机制）
5. 系统管理员是可信的（通过审计日志约束）

安全边界:
- 系统内部: 完全可信，数据明文传输
- API网关: 身份验证边界
- 外部API: 数据脱敏边界
```
