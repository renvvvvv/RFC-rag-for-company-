## 1. 超级管理员（Super Admin）

### 基本信息

| 项目 | 描述 |
|------|------|
| 姓名 | 王建国 |
| 职位 | IT 部门负责人 / 平台运维总监 |
| 年龄 | 40 岁 |
| 工作年限 | 15 年 |
| 所在行业 | 金融科技 / 制造业 / 大型国企 |

### 典型场景

- 负责系统私有化部署、模型配置、安全策略制定。
- 管理用户账号、用户组、权限体系。
- 监控系统运行状态，处理安全事件与审计需求。
- 向上级汇报系统可用性、合规性与使用数据。

### 核心目标

1. 确保系统稳定、安全、合规运行。
2. 最小化运维成本与故障恢复时间。
3. 满足企业审计与数据主权要求。
4. 快速响应业务部门账号与权限变更需求。

### 痛点

| 编号 | 痛点 | 影响 |
|------|------|------|
| P1 | 担心私有化部署配置复杂，缺少统一运维入口 | 高 |
| P2 | 用户权限变更频繁，手动维护效率低 | 高 |
| P3 | 缺乏完善的监控告警，故障发现滞后 | 中 |
| P4 | 审计日志分散，合规检查耗时 | 中 |
| P5 | 模型服务异常时缺少自动切换能力 | 中 |

### 优先级功能

| 优先级 | 功能 |
|--------|------|
| P0 | 统一系统管理后台（用户/组/权限/模型/审计） |
| P0 | Docker Compose / K8s 一键部署与回滚 |
| P1 | Prometheus + Grafana 监控告警 |
| P1 | 审计日志导出与合规报告 |
| P2 | 模型服务多地址容灾 |

### 典型语录

> “系统必须部署在我们自己的服务器上，数据不能出域，出了问题我能第一时间知道。”

## 2. 知识库管理员（KB Admin）

### 基本信息

| 项目 | 描述 |
|------|------|
| 姓名 | 李晓明 |
| 职位 | 部门知识管理员 / 文档中心负责人 |
| 年龄 | 32 岁 |
| 工作年限 | 8 年 |
| 所在行业 | 法律、咨询、研发、人力资源 |

### 典型场景

- 负责本部门知识库的创建、维护与内容治理。
- 审核成员上传的文档，确保内容准确、合规。
- 配置知识库权限，处理普通用户的访问申请。
- 通过评测工作台优化检索效果。

### 核心目标

1. 让部门知识资产可快速被找到和使用。
2. 确保不同角色只看到授权范围内的内容。
3. 减少重复回答和低效检索。
4. 持续提升问答准确率。

### 痛点

| 编号 | 痛点 | 影响 |
|------|------|------|
| P1 | 文档版本多，缺少统一版本管理与过期提醒 | 高 |
| P2 | 权限配置复杂，尤其是字段级/标签级权限 | 高 |
| P3 | 用户经常反馈“搜不到”，但不知如何优化 | 中 |
| P4 | 敏感文档需要额外审批与关键词过滤 | 中 |
| P5 | 批量上传大视频/大 PDF 解析失败率高 | 中 |

### 优先级功能

| 优先级 | 功能 |
|--------|------|
| P0 | 知识库 CRUD、文档批量上传与状态管理 |
| P0 | 细粒度权限配置（文档/字段/标签/关键词） |
| P1 | 评测工作台与 A/B 测试 |
| P1 | 敏感关键词管理与扫描 |
| P2 | 文档版本控制与过期提醒 |

### 典型语录

> “我不希望敏感合同内容被所有人看到，但普通员工查制度要方便快捷。”

| 项目 | 描述 |
|------|------|
| 姓名 | 张婷婷 |
| 职位 | 产品经理 / 销售 / 法务 / 研发工程师 |
| 年龄 | 28 岁 |
| 工作年限 | 4 年 |
| 所在行业 | 互联网、金融、制造、医药 |

### 典型场景

- 日常工作中需要频繁查找制度、规范、方案、合同、产品资料。
- 使用自然语言提问，快速获得答案与引用来源。
- 上传工作相关文档到授权知识库。
- 对答案质量进行反馈，帮助优化系统。

### 核心目标

1. 快速找到准确答案，减少信息检索时间。
2. 通过自然语言提问，降低搜索门槛。
3. 答案来源可追溯，便于核对与汇报。
4. 在权限范围内便捷上传和分享自己的文档。

### 痛点

| 编号 | 痛点 | 影响 |
|------|------|------|
| P1 | 传统搜索依赖关键词，找不到口语化表达的内容 | 高 |
| P2 | 答案没有引用来源，不敢直接用于工作 | 高 |
| P3 | 上传文档后不知道什么时候能被搜索到 | 中 |
| P4 | 多轮追问时上下文容易丢失 | 中 |
| P5 | 搜索结果包含大量无权限内容但无法查看 | 低 |

### 优先级功能

| 优先级 | 功能 |
|--------|------|
| P0 | 自然语言问答与引用溯源 |
| P0 | 全局/单知识库搜索 |
| P1 | 文档上传进度与状态可见 |
| P1 | 多轮对话与历史记录 |
| P2 | 答案点赞/点踩反馈 |

### 典型语录

> “我不想记文件名，只想问‘Q1 产品发布计划是什么’，然后直接拿到答案和出处。”

### 基本信息

| 项目 | 描述 |
|------|------|
| 姓名 | 陈志强 |
| 职位 | 外部顾问 / 合作伙伴 / 临时实习生 |
| 年龄 | 35 岁 |
| 工作年限 | 10 年 |
| 所在行业 | 咨询、审计、供应链、培训 |

### 典型场景

- 因项目需要临时访问企业的公开知识库。
- 仅查看被明确授权给访客的内容。
- 不使用敏感功能（如下载、上传、模型配置）。
- 访问结束后账号自动失效或被回收。

### 核心目标

1. 在最小权限范围内获取所需资料。
2. 不需要复杂培训即可使用搜索与问答。
3. 明确知道自己能看什么、不能看什么。

### 痛点

| 编号 | 痛点 | 影响 |
|------|------|------|
| P1 | 不清楚自己能访问哪些知识库 | 高 |
| P2 | 申请权限流程复杂，审批慢 | 中 |
| P3 | 界面功能过多，容易误操作 | 中 |
| P4 | 临时账号到期后无法自助续期 | 低 |

### 优先级功能

| 优先级 | 功能 |
|--------|------|
| P0 | 受限视角的搜索与问答 |
| P0 | 清晰的权限提示与申请入口 |
| P1 | 简化版界面（隐藏管理功能） |
| P2 | 临时账号生命周期管理 |

### 典型语录

> “我只是临时来查一些公开资料，界面越简单越好，别让我点错。”

## 6. 设计启示

基于以上画像，产品设计中应重点关注：

1. **权限即体验**：不同角色看到的功能菜单与数据范围必须严格区分。
2. **降低认知负担**：普通用户和访客需要简洁的搜索/问答入口，避免被管理功能干扰。
3. **可观测性优先**：Super Admin 需要一站式运维入口，避免在多系统间切换。
4. **效果可验证**：KB Admin 需要评测工具来持续优化检索与生成质量。
5. **反馈闭环**：所有用户都应有便捷的反馈通道，帮助系统迭代。
