## 1. 项目概述与目标

### 1.1 项目背景

随着企业知识资产日益多元化，传统的文本RAG系统已无法满足实际业务需求。企业内部沉淀的知识涵盖 **视频培训资料、会议录音、产品图片、技术文档、财务报表（Excel）、合同（PDF/Word）** 等多种模态。同时，企业级应用对 **数据安全、访问权限、合规审计** 有着极高的要求。

本方案旨在设计并构建一套 **企业级私有化多模态RAG知识库系统**，具备以下核心特征：

- **全模态覆盖**：支持视频、~~音频~~（待设计）、图片、PDF、Word、Excel、网页链接的统一管理与检索
- **细粒度权限**：从文件类型 → 单文档 → 文档内字段（Word/Excel单元格）→ 样本标签，四级权限控制
- **专用RAG流**：针对不同模态设计最优化的处理Pipeline（图片RAG、~~音频RAG~~【待设计】、视频RAG、文档RAG）
- **统一检索接口**：单一API入口，跨模态全局检索，所有文档参与召回
- **高效重排序**：集成用户提供的Re-rank模型，多阶段排序保障检索质量
- **评测闭环**：内置RAG评测工具，持续优化检索与生成效果

### 1.2 核心目标

| 维度 | 目标描述 |
|------|----------|
| **功能完备** | 覆盖11种+文件类型的上传、解析、索引、检索、生成全链路（音频RAG待设计） |
| **权限精细** | 支持四级权限（类型/文档/字段/标签），RBAC+ABAC混合模型 |
| **检索高效** | 百毫秒级跨模态检索响应，支持千万级文档规模 |
| **扩展灵活** | 统一API接口，可无缝对接外部LLM平台、Agent系统、业务应用 |
| **私有化安全** | 全栈私有化部署，数据不出域，支持审计追踪 |

### 1.3 设计原则

1. **模态原生保留**：视频、图片保留原始像素/帧信息参与检索，不做有损的纯文本转换
2. **权限前置过滤**：所有检索请求先经过权限过滤，再执行向量搜索，确保零泄露
3. **Pipeline可编排**：各模态RAG Pipeline模块化设计，支持动态插拔和参数调优
4. **统一与专用并存**：底层检索引擎统一，上层处理逻辑专用化
5. **渐进式实施**：MVP → 扩展 → 优化，分阶段验证价值

## 2. 行业调研与对标分析

### 2.1 企业级RAG发展趋势（2025-2026）

通过调研国内外主流企业级RAG平台和学术研究，总结出以下关键趋势：

**趋势一：多模态RAG成为标配**
- 文本-only RAG已无法满足企业需求，图片、视频、~~音频~~知识占比持续上升
- 行业主流方案采用 **"Separate Vector Stores + Late Fusion"** 架构：各模态独立索引，检索时并行召回、融合排序
- 视觉文档检索使用 **ColPali/ColQwen** 等布局感知嵌入模型，效果远超OCR+文本方案

**趋势二：混合检索是生产级标配**
- 单一向量检索召回率不足，企业级系统普遍采用 **BM25关键词 + Dense语义向量 + 元数据过滤 + Re-rank** 四层检索
- Reciprocal Rank Fusion（RRF）成为多路召回结果融合的标准算法
- 上下文感知的Cross-Encoder重排序显著提升Top-K精度

**趋势三：权限控制走向细粒度动态化**
- 从静态RBAC演进到 **RBAC + ABAC 混合模型**
- 当康数据等企业已申请基于AI分类分级的动态访问控制专利
- 文档标签成为权限控制的重要维度，支持自动化标签→权限映射

**趋势四：Agentic RAG崛起**
- RAG系统从被动检索向主动推理演进
- 支持Self-Correcting Retrieval、Multi-Step Reasoning、Tool Integration

### 2.2 主流方案对标

| 平台/方案 | 多模态支持 | 权限粒度 | 混合检索 | 私有化 | 可借鉴点 |
|-----------|-----------|----------|----------|--------|----------|
| RAGFlow | 文本为主 | 文档+标签级 | 是 | 支持 | 文档解析Pipeline、标签权限 |
| Dify | 有限 | RBAC+字段级 | 部分 | 支持 | 细粒度API权限、审批流 |
| MaxKB | 文本为主 | 资源级 | 是 | 支持 | 双重授权机制、模块化权限 |
| Mixpeek | 全模态 | 命名空间级 | 是 | 企业版 | 多模态统一检索Pipeline |
| Ragie | 音视频+文本 | 应用级 | 是 | SaaS | 音视频处理最佳实践（15s切片） |
| ColPali/ColQwen | 视觉文档 | 无 | 向量 | 开源 | 视觉文档检索SOTA方案 |

### 2.3 本方案差异化优势

相较于现有方案，本系统的独特设计在于：

1. **四级权限穿透**：业界少有的从文件类型 → 文档 → 字段 → 标签的完整权限链
2. **字段级权限**：Word/Excel支持到段落/单元格/工作表级别的可见性控制
3. **统一跨模态全局检索**：单次查询同时检索所有模态，无需分 modality 发起请求
4. **评测工具内置**：前端直接集成RAG评测工作流，支持A/B测试和效果对比

### 4.3 评测工作台设计

评测工作台用于对RAG系统的检索质量和生成质量进行系统化评估。

#### 4.3.1 评测功能模块

| 模块 | 功能描述 |
|------|----------|
| **评测集管理** | 创建/导入评测问答对(QA Pair)，支持CSV/JSON导入 |
| **检索评测** | 评估检索召回率、精确率、MRR、NDCG等指标 |
| **生成评测** | 评估答案相关性、忠实度、答案正确性 |
| **A/B对比** | 对比不同配置（Chunk大小、Top-K、Re-rank模型）的效果差异 |
| **Bad Case分析** | 自动聚类失败案例，定位问题根因 |
| **趋势报告** | 历史评测结果趋势图，追踪系统优化效果 |

#### 4.3.2 评测指标体系

```
检索质量指标：
+-- Recall@K      - Top-K召回率
+-- Precision@K   - Top-K精确率
+-- MRR           - 平均倒数排名
+-- NDCG@K        - 归一化折损累计增益
+-- Hit Rate      - 命中率

生成质量指标：
+-- Answer Relevance    - 答案相关性（LLM-as-Judge）
+-- Faithfulness        - 忠实度（是否基于检索内容）
+-- Context Precision   - 上下文精确度
+-- Context Recall      - 上下文召回率
+-- Answer Correctness  - 答案正确性
```

## 6. 多层级权限控制体系

权限控制是本系统的核心差异化能力。我们采用 **RBAC（基于角色）+ ABAC（基于属性）混合模型**，实现从粗到细的四级权限穿透。

### 6.1 权限模型总览

```
权限控制层级（从粗到细）：

Level 1: 文件类型权限
    "用户A可以访问【视频】和【PDF】类型的文档，但不能访问【Excel】"
    
Level 2: 文档级别权限
    "用户A可以访问文档D001、D003，不能访问D002"
    
Level 3: 字段/内容级别权限（Word/Excel专用）
    "用户A可以看到Excel表'工资表'的'姓名'和'部门'列，但看不到'薪资'列"
    "用户A可以看到Word文档的前3节，第4节'机密条款'不可见"
    
Level 4: 标签级别权限
    "用户A可以访问标签为'公开'和'内部'的文档，但不能访问'机密'标签的文档"
    "用户A可以访问打了'样本标签-A'的文档片段"
```

#### 6.3.3 Level 3: 字段/内容级别权限（核心创新）

这是本系统最具差异化的权限设计，支持对 **Word段落** 和 **Excel单元格/工作表/列** 进行细粒度控制。

**核心思路：**
- 文档解析时，不仅提取文本内容，还记录 **结构化位置信息**
- 索引时，每个Chunk携带其 **来源位置元数据**
- 检索时，根据用户权限过滤掉无权限位置的Chunk
- 生成时，仅使用有权限的Chunk作为上下文

**Word文档字段权限：**

```python
class WordFieldPermission(BaseModel):
```

**字段权限在检索中的实现：**

```
文档解析阶段:
Word文档 -> 解析为段落列表 -> 每个段落记录:
    - paragraph_id
    - section_id  
    - heading_path (如: "1.合同总则/1.1定义")
    - text_content
    - bounding_box (页面位置)

Excel文档 -> 解析为单元格列表 -> 每个单元格记录:
    - sheet_name
    - cell_address (如: "A1", "B3")
    - column_name (如: "姓名")
    - row_number
    - value
    - data_type

索引阶段:
每个Chunk的metadata中携带:
    - source_positions: [{段落范围/单元格范围}]
    - section_headings: [章节标题列表]
    - sheet_name: 工作表名
    - column_names: [涉及的列名]

检索阶段:
1. 召回Top-K候选Chunk
2. 对每个Chunk，检查其source_positions是否在用户权限范围内
3. 过滤掉越权Chunk
4. 如果某个文档的所有Chunk都被过滤，则该文档对用户"不可见"
```

#### 6.3.4 Level 4: 标签级别权限

标签权限用于控制用户对打了特定标签的文档/文档片段的访问。

**标签权限模型：**

```python
class TagPermission(BaseModel):
    """标签权限"""
    tag_id: str
    tag_name: str
    tag_category: str       # 标签分类: "安全等级" / "业务部门" / "样本标签" / ...

    tag_id: str
    target_type: str        # "document" / "chunk" / "paragraph" / "cell"
    target_id: str

    """用户-标签权限映射"""
    user_id: str
    role_id: str
    allowed_tags: List[str]     # 允许访问的标签列表
    denied_tags: List[str]      # 明确禁止的标签列表

# 示例: 安全等级标签权限
# 标签定义:
#   "公开"(public) / "内部"(internal) / "机密"(confidential) / "绝密"(top_secret)
#
# 用户权限:
#   实习生: ["公开"]
#   普通员工: ["公开", "内部"]
#   管理层: ["公开", "内部", "机密"]
#   高管: ["公开", "内部", "机密", "绝密"]

# 示例: 样本标签权限（业务场景）
# 对文档中的特定片段打"样本标签"，用于AI训练数据管理
#   "样本标签-A": 可用于模型训练
#   "样本标签-B": 仅用于评测
#   "样本标签-C": 敏感数据，不可用于训练
```

**标签权限在检索中的实现：**

```python
def filter_chunks_by_tag_permission(chunks: List[Chunk], user: User) -> List[Chunk]:
    """
    根据用户标签权限过滤Chunk
    
    规则:
    1. Chunk的标签集合必须与用户allowed_tags有交集
    2. Chunk不能包含任何denied_tags
    3. 如果Chunk无标签，默认允许访问（除非系统配置为拒绝无标签）
    """
    user_allowed = set(user.tag_permissions.allowed_tags)
    user_denied = set(user.tag_permissions.denied_tags)
    
    filtered = []
    for chunk in chunks:
        chunk_tags = set(chunk.tags)
        
        # 检查是否有禁止标签
        if chunk_tags & user_denied:
            continue
            
        # 检查是否有允许标签（或无标签）
        if not chunk_tags or (chunk_tags & user_allowed):
            filtered.append(chunk)
            
    return filtered
```

### 6.4 权限缓存与性能优化

权限检查在每次检索时都会执行，需要保证高性能：

```
权限缓存策略:

1. 用户权限缓存 (Redis, TTL=5分钟)
   Key: "user_perm:{user_id}" 
   Value: 序列化后的完整权限配置
   
2. 文档权限缓存 (Redis, TTL=10分钟)
   Key: "doc_perm:{doc_id}:{user_id}"
   Value: ALLOW / DENY / PARTIAL
   
3. 字段权限缓存 (Redis, TTL=10分钟)
   Key: "field_perm:{doc_id}:{user_id}"
   Value: 允许的字段范围JSON
   
4. 标签权限缓存 (Redis, TTL=5分钟)
   Key: "tag_perm:{user_id}"
   Value: 允许/禁止的标签集合

5. 组合权限位图 (高性能场景)
   使用Redis Bitmap预计算每个用户对每个文档的最终权限
   Key: "perm_bitmap:{user_id}"
   位偏移: doc_id哈希值
   位值: 0=拒绝, 1=允许
```

## 7. 多模态RAG Pipeline设计

这是本系统的技术核心。针对不同模态设计专用的处理Pipeline，同时通过统一的向量化接口和检索接口对外提供服务。

### 7.1 Pipeline总体架构

```
+--------------------------------------------------------------------------+
|                         统一入口: Ingest Service                         |
|                    (接收文件 -> 格式识别 -> 路由分发)                      |
+--------------------------------------------------------------------------+
       |              |              |              |              |
       v              v              v              v              v
+----------+   +----------+   +----------+   +----------+   +----------+
| 文档     |   | 图片     |   | ~~音频~~ |   | 视频     |   | 链接     |
| Pipeline |   | Pipeline |   | Pipeline |   | Pipeline |   | Pipeline |
+----------+   +----------+   +----------+   +----------+   +----------+
       |              |              |              |              |
       v              v              v              v              v
+--------------------------------------------------------------------------+
|                      统一向量化层 (Embedding Service)                    |
|               (用户提供的Embedding模型 + 多模态适配器)                    |
+--------------------------------------------------------------------------+
       |
       v
+--------------------------------------------------------------------------+
|                      统一索引层 (Index Service)                          |
|           (分模态向量集合 -> 统一元数据Schema -> 批量入库)                 |
+--------------------------------------------------------------------------+
```

### 7.2 各模态Pipeline详细设计

#### 7.2.1 文档RAG Pipeline（PDF/Word/TXT）

```
输入: PDF / Word / TXT 文件
  |
  v
+---------------------------------------------------+
| 1. 文档解析                                        |
|   - PDF: 使用pdfplumber/unstructured提取文本+布局   |
|   - Word: 使用python-docx提取段落+表格+样式         |
|   - TXT: 直接读取                                  |
|   输出: 结构化文档对象                              |
+---------------------------------------------------+
  |
  v
+---------------------------------------------------+
| 2. 结构化提取                                      |
|   - 识别标题层级 (H1-H6)                           |
|   - 提取表格 (保留行列关系)                         |
|   - 识别列表、代码块等特殊元素                      |
|   输出: 带位置信息的元素列表                        |
+---------------------------------------------------+
  |
  v
+---------------------------------------------------+
| 3. 智能分块 (Chunking)                             |
|   策略选择:                                         |
|   a) 固定大小分块: 512 tokens / 256 overlap        |
|   b) 语义分块: 按段落/章节边界分割                  |
|   c) 结构感知分块: 保持表格/列表完整性              |
|   d) 混合策略: 优先语义边界，超长时强制切割          |
|   输出: Chunk列表，每个Chunk携带位置元数据           |
+---------------------------------------------------+
  |
  v
+---------------------------------------------------+
| 4. 向量化                                          |
|   - 用户提供的Embedding模型生成Dense Vector         |
|   - 同时生成Sparse Vector (BM25关键词权重)          |
|   输出: (dense_vector, sparse_vector, metadata)    |
+---------------------------------------------------+
  |
  v
+---------------------------------------------------+
| 5. 索引入库                                        |
|   - Dense Vector -> 向量数据库 (Milvus Collection)  |
|   - Sparse Vector -> 倒排索引 (Elasticsearch/Meilisearch)
|   - Metadata -> 关系数据库 (PostgreSQL)             |
+---------------------------------------------------+
```

**关键设计点：**

| 设计点 | 方案 | 理由 |
|--------|------|------|
| PDF解析 | unstructured + 自研表格识别 | unstructured对复杂PDF支持好，自研增强表格 |
| 分块策略 | 结构感知语义分块 | 保留文档结构，提升检索上下文质量 |
| 表格处理 | 每个表格生成独立Chunk | 表格信息密度高，单独索引效果更好 |
| 双向量索引 | Dense + Sparse并存 | Dense语义匹配 + Sparse关键词精确匹配 |

### 8.5 结果融合（Fusion）

多路召回后，需要将不同路径的结果融合为统一的候选列表。

#### 8.5.1 Reciprocal Rank Fusion (RRF)

```
RRF公式:
  score(d) = Σ(1 / (k + rank_i(d)))
  
其中:
  - rank_i(d): 文档d在第i个召回路径中的排名
  - k: 常数，通常取60
  
RRF特点:
  - 不需要校准不同路径的分数尺度
  - 排名靠前的高权重
  - 简单高效，业界标准
```

```python
def reciprocal_rank_fusion(results_per_path: Dict[str, List[Doc]], k: int = 60) -> List[Doc]:
    """
    RRF融合多路召回结果
    """
    scores = defaultdict(float)
    doc_map = {}
    
    for path_name, docs in results_per_path.items():
        for rank, doc in enumerate(docs, start=1):
            doc_map[doc.id] = doc
            scores[doc.id] += 1.0 / (k + rank)
    
    # 按融合分数排序
    sorted_docs = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return [doc_map[doc_id] for doc_id, _ in sorted_docs]
```

### 8.6 重排序（Re-ranking）

重排序是提升检索质量的关键环节。本方案采用 **两阶段重排序** 策略。

#### 8.6.1 第一阶段: 轻量Cross-Encoder预排序

```
目的: 快速剔除明显不相关的候选
输入: RRF融合后的Top-100候选
模型: 轻量级Cross-Encoder (如bge-reranker-base)
输出: 重排序后的Top-50候选
```

#### 8.6.2 第二阶段: 用户提供的Re-rank模型精排

```
目的: 利用用户专用模型做最终精排
输入: 第一阶段Top-50候选
模型: 用户提供的Re-rank模型
输出: 最终Top-K (通常K=5~10) 结果
```

```python
class TwoStageReranker:
    def __init__(self):
        self.stage1_model = load_model("bge-reranker-base")  # 轻量模型
        self.stage2_model = load_user_reranker()              # 用户模型
    
    async def rerank(self, query: str, candidates: List[Chunk], top_k: int = 10) -> List[Chunk]:
        # Stage 1: 快速预排序
        if len(candidates) > 50:
            stage1_scores = self.stage1_model.predict([(query, c.text) for c in candidates])
            candidates = sorted(
                zip(candidates, stage1_scores),
                key=lambda x: x[1], reverse=True
            )[:50]
            candidates = [c for c, _ in candidates]
        
        # Stage 2: 精排
        stage2_scores = self.stage2_model.predict([
            (query, self._build_input(c)) for c in candidates
        ])
        
        # 最终排序
        ranked = sorted(
            zip(candidates, stage2_scores),
            key=lambda x: x[1], reverse=True
        )[:top_k]
        
        return [c for c, _ in ranked]
    
    def _build_input(self, chunk: Chunk) -> str:
        """构建Re-rank模型的输入文本"""
        # 组合Chunk的多模态信息
        parts = [chunk.text]
        if chunk.metadata.get("visual_description"):
            parts.append(f"[图片描述] {chunk.metadata['visual_description']}")
        if chunk.metadata.get("transcript"):
            parts.append(f"[转录] {chunk.metadata['transcript']}")
        return " | ".join(parts)
```

### 8.7 权限后过滤

重排序后，还需要做一次 **字段级权限过滤**：

```python
def apply_field_level_filter(chunks: List[Chunk], user: User) -> List[Chunk]:
    """
    字段级权限后过滤
    这一步在重排序后执行，因为重排序模型需要看到完整的候选
    """
    filtered = []
    for chunk in chunks:
        doc_id = chunk.doc_id
        
        # 检查文档类型权限
        if not has_file_type_permission(user, chunk.modality):
            continue
        
        # 检查字段级权限（Word/Excel）
        if chunk.modality in ["DOCUMENT", "EXCEL"]:
            if not has_field_permission(user, doc_id, chunk.positions):
                continue
        
        # 检查标签权限
        if not has_tag_permission(user, chunk.tags):
            continue
        
        filtered.append(chunk)
    
    return filtered
```

```python
def build_generation_context(query: str, chunks: List[Chunk]) -> str:
    """
    构建带引用的生成上下文
    """
    context_parts = []
    for i, chunk in enumerate(chunks, 1):
        # 添加来源引用标记
        source_ref = f"[{i}]"
        
        # 根据模态组装内容
        if chunk.modality == "image":
            content = f"{source_ref} [图片] {chunk.text}\n图片URL: {chunk.metadata['image_url']}"
        elif chunk.modality == "audio":
            content = f"{source_ref} [音频] {chunk.text}\n时间: {chunk.metadata['time_range']}"  # 音频RAG待设计，当前仅存储
        elif chunk.modality == "video":
            content = f"{source_ref} [视频片段] {chunk.text}\n时间: {chunk.metadata['time_range']}"
        elif chunk.modality == "excel":
            content = f"{source_ref} [表格数据] {chunk.text}\n工作表: {chunk.metadata['sheet_name']}"
        else:
            content = f"{source_ref} {chunk.text}"
        
        context_parts.append(content)
    
    return "\n\n---\n\n".join(context_parts)
```

### 10.2 关键表结构

#### users（用户表）

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    role_id UUID REFERENCES roles(id),
    department_id UUID REFERENCES departments(id),
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### knowledge_bases（知识库表）

```sql
CREATE TABLE knowledge_bases (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(100) NOT NULL,
    description TEXT,
    owner_id UUID REFERENCES users(id),
    config JSONB DEFAULT '{}',  -- embedding配置、rerank配置等
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### documents（文档表）

```sql
CREATE TABLE documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    kb_id UUID REFERENCES knowledge_bases(id),
    filename VARCHAR(255) NOT NULL,
    file_type VARCHAR(50) NOT NULL,  -- document, pdf, excel, image, audio, video, link
    file_size BIGINT,
    mime_type VARCHAR(100),
    storage_key VARCHAR(255),  -- MinIO中的对象key
    status VARCHAR(20) DEFAULT 'pending',  -- pending, processing, indexed, failed
    processing_info JSONB DEFAULT '{}',  -- 处理进度、错误信息等
    metadata JSONB DEFAULT '{}',  -- 自定义元数据
    created_by UUID REFERENCES users(id),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### chunks（文档片段表）

```sql
CREATE TABLE chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    doc_id UUID REFERENCES documents(id),
    content TEXT NOT NULL,
    modality VARCHAR(50) NOT NULL,
    chunk_index INTEGER,  -- 在文档中的顺序
    
    -- 位置信息（JSONB，根据模态不同结构不同）
    position_info JSONB DEFAULT '{}',
    -- 示例: 
    -- Word: {"paragraph_range": [10, 15], "section_path": "1.2.3"}
    -- Excel: {"sheet_name": "Sheet1", "cell_range": "A1:C10", "column_names": ["姓名", "部门", "薪资"]}
    -- Audio: {"time_range": {"start": 135.0, "end": 148.5}}
    -- Video: {"time_range": {"start": 330.0, "end": 345.0}, "frame_indices": [3300, 3350]}
    -- Image: {"bounding_box": {"x": 0, "y": 0, "w": 1920, "h": 1080}}
    
    -- 向量引用（实际向量存在Milvus中）
    vector_id VARCHAR(100),  -- Milvus中的主键
    
    -- 用于关键词检索的TSV向量（可选）
    content_tsv TSVECTOR,
    
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 全文检索索引
CREATE INDEX idx_chunks_tsv ON chunks USING GIN(content_tsv);
```

#### permissions（权限表）

## 16. 风险与应对策略

| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| **Embedding/Re-rank模型接入复杂** | 高 | 中 | 设计标准接口抽象层，提供适配器模板和示例代码 |
| **字段级权限性能问题** | 高 | 中 | 预计算权限位图，Redis缓存，异步权限校验 |
| **视频处理资源消耗大** | 中 | 中 | 当前阶段仅处理画面（无ASR），GPU消耗降低；限制上传大小和时长，队列限流 |
| **多模态检索效果不佳** | 高 | 中 | 先做MVP验证效果，预留模型切换能力，建立评测基准 |
| **权限系统过于复杂** | 中 | 中 | 分阶段实现，先文档级再字段级；提供默认配置模板 |
| **向量数据库扩展性** | 中 | 低 | 选用Milvus分布式版本，预留分片策略 |
| **大文件上传稳定性** | 中 | 中 | 分片上传+断点续传+MD5校验 |
| **跨模态融合效果** | 中 | 中 | RRF作为保底方案，支持A/B测试对比不同融合策略 |
