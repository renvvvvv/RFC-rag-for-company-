## 五、未修复/新发现的问题

### 5.1 Hybrid 模式超时（新增）

| query | 模式 | 现象 | 影响 |
| --- | --- | --- | --- |
| UPS5000-E 系列有哪些型号 | hybrid | 60s 超时，0 结果 | Hit@5=False（仅 hybrid） |
| 技术参数 | hybrid | 60s 超时，0 结果 | Hit@5=False（仅 hybrid） |

**根因推测**：hybrid 链路包含 BM25 + 向量检索 + RRF + rerank，rerank（qwen3-rerank）调用耗时较长。加上 D40 文档 chunk 数大（386），Top-K 扩大后 RRF 合并阶段计算量也增加。

**建议**：
- hybrid 模式客户端超时阈值提高到 90s
- 或为 hybrid 模式设置更小的候选 Top-K（rerank 前 20 → 10）
- 或将 rerank 调用改为异步 / 失败降级

### 5.2 rerank_score 字段未回传（修复中遗留）

- 所有 42 次返回的 `sources[].rerank_score` 均为 `null`
- 之前 BadCase 报告 §2.3 提到 "Rerank 未改变排序（score = rerank_score）"，本次测试中 `rerank_score` 字段直接为 null
- **影响**：无法用 rerank_score 验证 rerank 排序是否改变；`score` 是 RRF 合并后的综合分

**建议**：检查 `retrieval_service.py` 中 rerank 结果合并逻辑，确认 `rerank_score` 字段是否被正确写入 `Source` 对象返回。

### 5.3 D40 仍主导 Hit@1（已知结构性问题）

- D40（386 chunks）占 94.8%，对通用查询（"模块重量""产品规格""如何启动 UPS"），D40 的相关 chunk 相似度仍排第一
- Q8/Q9/Q14 期望 D50 出现在 Top-1，但实际 D40 的更宽泛的 chunk 排在前面
- **这不是 BadCase 报告中的语义偏移问题**（D50 已正确进入 Top-5，per-doc 去重生效），而是"文档量级悬殊"导致的剩余偏差

**建议（可选）**：
- 实施 BadCase 报告 §5.2 方案 A（文档级 MMR，每文档最多 2 条）→ 进一步保证 D50 的位置
- 或在 RRF 合并时按 `log(total_chunks / doc_chunk_count)` 加权降权大文档

### 5.4 LLM 回答生成持续 503

- 42 次 chat 调用中 LLM 回答部分全部返回 "抱歉，回答生成服务暂时不可用，以下是检索到的相关片段..."
- **本评测只关注 sources 部分**，故不受影响；但生产环境用户体验受损
- **建议**：检查 LLM 服务（外部 LLM / 自部署模型）连通性 / API Key / 限流

### 6.2 修复验证结论

**之前的三类修复（BM25 ILIKE fallback / Top-K 扩大 / per-doc 去重）全部生效**：

1. **BM25 ILIKE fallback** 解决了 BC-1/BC-3/BC-4 的 keyword 模式 0 命中问题（中文+英文混合 token / 长中文 token）
2. **Top-K 扩大 + per-doc 去重** 解决了 BC-2/BC-5 的 D50/D60 被 D40 淹没问题（hybrid 模式也能让 D50 的目标 chunk 排到 Top-1）
3. **rerank 诊断** 待进一步确认（rerank_score 字段未回传，无法量化验证）
