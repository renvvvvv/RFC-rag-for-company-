## 1. 企业知识管理员 — 李敏

### 基础信息

| 项目 | 内容 |
|------|------|
| 姓名 | 李敏 |
| 年龄 | 34 岁 |
| 职位 | 企业知识管理部负责人 / 部门 KB Admin |
| 部门 | 研发中心知识管理部 |
| 工作年限 | 10 年 |
| 安全级别 | L3 |

### 日常工作

- 负责研发中心多个知识库的创建、分类和维护。
- 制定文档命名规范、标签体系和密级标准。
- 为不同项目组、职能部门分配知识库访问权限。
- 定期审查文档过期、失效和敏感内容。
- 收集用户反馈，优化检索效果和文档组织方式。

### 核心诉求

1. **知识库管理便捷**：能够快速创建、配置和归档知识库。
2. **权限配置灵活**：支持按部门、项目、角色、文档、字段等多维度授权。
3. **文档生命周期可控**：上传、解析、索引、禁用、删除、重新处理全流程可管理。
4. **效果可量化**：通过评测工作台了解知识库检索质量，持续优化。
5. **合规审计留痕**：所有权限变更、文档操作、敏感访问都有审计日志。

### 典型使用场景

> 李敏收到产品线要求，需要为"Q3 新品发布"建立独立知识库。她登录系统后创建知识库，设置可见范围为"产品部+市场部"，配置默认分块策略，批量上传产品资料、竞品分析、培训视频。随后她为财务相关 Excel 表格设置字段级权限，隐藏"成本"列，确保只有财务组可见。最后她在评测工作台导入 50 条测试问题，验证检索召回率。

### 痛点

| 痛点 | 影响 | 优先级 |
|------|------|--------|
| 批量授权操作繁琐 | 管理效率低 | 高 |
| 权限继承规则不透明 | 容易配置错误，存在泄露风险 | 高 |
| 大视频解析进度不可见 | 用户体验差，重复咨询 | 中 |
| 缺乏知识库使用热度分析 | 难以判断内容价值 | 中 |

## 2. 普通员工 — 王强

### 基础信息

| 项目 | 内容 |
|------|------|
| 姓名 | 王强 |
| 年龄 | 28 岁 |
| 职位 | 销售工程师 |
| 部门 | 销售部华东区 |
| 工作年限 | 4 年 |
| 安全级别 | L1 |

### 日常工作

- 拜访客户前快速查询产品资料、报价策略、合同模板。
- 在客户现场回答技术问题，需要准确引用官方文档。
- 学习新产品知识和销售话术，观看培训视频。
- 提交常见问题和资料缺失反馈。

### 核心诉求

1. **搜索快速准确**：用自然语言提问即可找到答案，不用记文件名或关键词。
2. **答案可信可溯源**：每条答案都能查看来自哪份文档、哪一页或哪个视频片段。
3. **操作简单**：界面清晰，无需培训即可使用。
4. **权限自然**：只看到自己被授权的内容，不会因为权限问题频繁申请。
5. **支持多模态**：能搜到 PDF、Excel 表格、产品图片、培训视频等多种资料。

### 典型使用场景

> 王强明天要去拜访一家制造业客户，客户在意的场景是"产线质检"。他在搜索框输入"质检方案包含哪些硬件？"，系统从"产品资料库"召回相关 PDF 段落、产品图片和案例视频。他点击答案下方的引用来源，打开 PDF 原文第 12 页，确认信息无误后复制要点到客户方案中。随后他为该答案点赞，帮助系统优化排序。

### 痛点

| 痛点 | 影响 | 优先级 |
|------|------|--------|
| 答案偶尔不准确或" hallucination " | 误导客户决策 | 高 |
| 找不到最新的产品资料 | 使用过期信息 | 高 |
| 搜索历史无法快速复用 | 重复输入相同问题 | 中 |
| 移动端无法方便查看 | 外出场景受限 | 中 |

## 3. 系统管理员 — 张涛

### 基础信息

| 项目 | 内容 |
|------|------|
| 姓名 | 张涛 |
| 年龄 | 38 岁 |
| 职位 | IT 运维与安全负责人 / Super Admin |
| 部门 | 信息技术部 |
| 工作年限 | 14 年 |
| 安全级别 | L4 |

### 日常工作

- 负责系统私有化部署、升级、扩容和故障处理。
- 管理用户账号、角色、用户组和全局权限策略。
- 配置 Embedding、Re-rank、LLM 等模型服务。
- 监控系统运行状态、资源使用和告警。
- 配合安全审计，导出审计日志和访问记录。
- 制定备份、回滚和灾难恢复方案。

### 核心诉求

1. **部署灵活**：支持 Docker Compose 和 Kubernetes Helm 两种部署形态。
2. **安全可控**：数据不出域、密钥不泄露、权限零漏洞、操作全审计。
3. **可观测性强**：日志、指标、告警、Dashboard 一应俱全。
4. **易于运维**：健康检查、自动扩缩容、滚动更新、备份恢复机制完善。
5. **合规支持**：满足等保、数据安全法、行业合规要求。

### 典型使用场景

> 张涛收到业务方上线通知，需要在生产环境部署 RAG 系统。他选择 Kubernetes Helm 方案，先生成 `secrets.yaml` 并替换所有默认密码，执行 `bash upgrade.sh rag-system rag-system`。随后他配置 Ingress + TLS，将 Alertmanager 告警路由到企业飞书 webhook。上线前他运行 `scripts/agent_api_harness.py` 验证全部接口，确认 73 PASS / 2 SKIP / 0 FAIL 后签字通过。

### 痛点

| 痛点 | 影响 | 优先级 |
|------|------|--------|
| 全量模式资源占用高 | 低配环境无法部署 | 高 |
| 默认配置存在安全风险（默认密码） | 上线前遗漏会导致安全事件 | 高 |
| Alertmanager 默认接收人为 placeholder | 告警无法触达 | 高 |
| K8s Startup Probe 未配置 | 启动异常时难以发现 | 中 |
| 缺乏运营数据看板 | 无法量化业务价值 | 中 |

## 4. 用户画像汇总

| 维度 | 李敏（知识管理员） | 王强（普通员工） | 张涛（系统管理员） |
|------|-------------------|------------------|-------------------|
| **核心目标** | 管好知识、控好权限、持续优化 | 快速找到可信答案 | 安全稳定部署运维 |
| **高频功能** | 知识库管理、权限配置、评测工作台 | 搜索、问答、查看引用 | 用户管理、模型配置、监控告警 |
| **关键痛点** | 批量授权、权限继承透明度 | 答案准确性、资料时效性 | 资源占用、默认安全配置 |
| **安全级别** | L3 | L1 | L4 |
| **主要关注点** | 效果、合规、效率 | 体验、准确、便捷 | 安全、稳定、可观测 |

## 5. 对产品设计的启示

1. **为知识管理员提供批量操作和权限规则可视化工具**，降低管理复杂度。
2. **为普通员工强化答案引用和反馈机制**，建立对 AI 答案的信任。
3. **为系统管理员提供一键安全加固向导**，避免默认配置风险。
4. **三个角色的工作台应差异化展示**：员工侧聚焦搜索问答，管理员侧聚焦配置监控，KB Admin 侧聚焦知识库运营。
