## 2. 业务背景

### 2.1 企业知识管理现状

1. **知识载体多样化**：制度文件（PDF/Word）、财务数据（Excel）、产品图片、培训视频、网页链接等并存。
2. **知识分布碎片化**：资料散落在邮件、网盘、IM、本地电脑、业务系统中，缺乏统一入口。
3. **检索方式落后**：传统关键词搜索无法理解语义，用户需要多次尝试才能定位信息。
4. **权限管理粗放**：多数系统仅支持文档级权限，无法满足字段级、标签级、关键词级的细粒度控制。
5. **AI 答案不可信**：通用大模型生成内容存在"幻觉"，且无法说明答案来源，企业不敢直接使用。

### 2.2 行业合规与安全要求

- **数据主权**：金融、政务、制造等行业要求核心数据不出域。
- **权限最小化**：敏感信息（财务、人事、战略）必须按角色、部门、项目精确控制。
- **审计留痕**：登录、上传、检索、权限变更等操作需要完整记录。
- **DevSecOps**：上线前需通过 SAST、SCA、DAST、容器扫描、SBOM 等安全检查。

### 2.3 技术趋势

- RAG（检索增强生成）成为企业落地大模型的主流路径。
- 多模态 RAG（图片、视频、表格）需求快速增长。
- 私有化部署与模型可控性成为 ToB 客户决策关键。
- 权限感知检索与上下文压缩是 enterprise RAG 的差异化能力。

## 3. 用户场景

### 3.1 场景一：研发知识库检索

**用户**：研发工程师  
**问题**："Q3 产品规划中，服务端 API 的鉴权方式变更了吗？"  
**期望**：系统从"研发技术规范库"召回相关 API 文档、发布说明，生成带引用的答案。

### 3.2 场景二：销售赋能问答

**用户**：销售工程师  
**问题**："制造业客户的质检方案包含哪些硬件？"  
**期望**：系统召回产品资料、案例视频、竞品对比，给出结构化回答并标注来源。

### 3.3 场景三：财务数据权限化访问

**用户**：财务人员 / 部门经理  
**问题**："销售部 Q2 的销售额排名前三的产品是什么？"  
**期望**：系统按用户级别和字段级权限，仅返回有权限的列和行数据。

### 3.4 场景四：培训视频切片检索

**用户**：新员工  
**问题**："新员工入职培训中关于信息安全的内容在哪里？"  
**期望**：系统从视频中召回关键帧和时间戳，定位到相关片段。

### 3.5 场景五：合规审计

**用户**：安全审计员  
**问题**："上个月哪些用户访问了机密标签的文档？"  
**期望**：系统提供完整审计日志，支持按时间、用户、操作类型筛选导出。

## 4. 功能需求汇总

### 4.1 知识库管理

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 知识库 CRUD | P0 | 创建、编辑、删除、列表、统计 |
| 可见范围配置 | P0 | 私有 / 部门 / 公开 |
| 分块策略配置 | P0 | chunk_size、overlap、默认模态 |
| 文档重新处理 | P0 | 支持更新后重新解析索引 |

### 4.2 文档摄取

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 文档解析（PDF/Word/TXT/Markdown） | P0 | 标题层级、表格、图片引用 |
| Excel/CSV 解析 | P0 | 表头识别、列/行/区域 Chunk |
| 图片解析 | P0 | OCR + Vision 描述 + 双向量 |
| 视频解析 | P1 | 关键帧提取、时序索引 |
| 链接抓取 | P1 | URL 正文抓取、增量更新 |
| 音频解析 | P2 | ASR 转录、时间戳对齐 |

### 4.3 检索与问答

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 语义检索 | P0 | Dense 向量检索 |
| 关键词检索 | P0 | BM25 / Sparse 检索 |
| 混合检索与 RRF 融合 | P0 | 多路召回结果融合 |
| 三种检索模式切换 | P0 | hybrid / semantic / keyword |
| 跨知识库/跨模态检索 | P0 | 统一检索入口 |
| 单库问答 | P0 | 基于知识库生成答案 |
| 多轮对话 | P0 | conversation_id 上下文管理 |
| 流式/非流式输出 | P0 | SSE 流式支持 |
| 引用溯源 | P0 | chunk_id、doc_id、页码/时间戳 |
| 答案反馈 | P0 | 点赞/点踩 |
| 搜索历史 | P0 | 用户历史检索记录 |

## 5. 非功能需求汇总

### 5.1 性能

| 指标 | 目标 |
|------|------|
| 检索 P99 延迟 | ≤ 2s |
| 检索平均延迟 | ≤ 800ms |
| 问答生成 P99 延迟 | ≤ 6.5s |
| 并发用户 | ≥ 100 |
| 文档索引延迟 | < 10MB 文档 2 分钟内可检索 |

### 5.2 可用性与可靠性

| 指标 | 目标 |
|------|------|
| 服务可用性 | ≥ 99.9%（生产 K8s） |
| 数据持久化 | PostgreSQL 主从 + Milvus/MinIO 多副本 |
| 备份策略 | PostgreSQL 每日全量 + WAL 实时；Milvus 每周全量 |
| 故障恢复 RTO | PostgreSQL < 1min；Milvus < 30s |

### 5.3 安全性

| 指标 | 目标 |
|------|------|
| 权限零泄露 | 100% |
| 敏感词拦截 | 按 L0-L4 生效 |
| DevSecOps | CI 集成 SAST/SCA/容器扫描/SBOM |
| Secret 管理 | 不在 Git 中提交 |

### 5.4 可观测性

| 指标 | 目标 |
|------|------|
| 监控覆盖 | Prometheus + Grafana 4 套 Dashboard |
| 告警规则 | 7 类核心告警 |
| 日志规范 | JSON 结构化 |

### 5.5 部署与扩展

| 指标 | 目标 |
|------|------|

## 6. 调研结论

### 6.1 核心结论

1. 企业对 RAG 的核心诉求是"私有化、可管控、可追溯"，而非单纯的问答能力。
2. 五级权限穿透、多模态 ingestion、引用溯源是 ToB 客户的关键差异化诉求。
3. 效果可量化（评测工作台）和安全可审计是签单和合规的必备条件。
4. 轻量部署模式可显著降低 POC 和中小规模客户的准入门槛。

### 6.2 建议产品边界

- **MVP 必须包含**：知识库、文档/Excel/图片摄取、混合检索、问答、五级权限、关键词拦截、模型配置、API Key、评测工作台、Docker Compose 部署。
- **P1 补齐**：视频/链接完整 Pipeline、权限管理端批量操作、任务中心实时进度、告警通道替换。
- **P2 远期**：知识图谱、IM 集成、Agentic RAG、多语言 UI、移动端适配。

### 6.3 下一步行动

1. 以 MVP 范围锁定 Sprint 1-6 的开发任务（详见 `docs/deliverables/sprint_plan.md`）。
2. 按 P0/P1/P2 分组管理需求变更，避免 MVP 范围蔓延。
3. 在客户交付前完成 `docs/deliverables/go_live_checklist.md` 全部检查项。
