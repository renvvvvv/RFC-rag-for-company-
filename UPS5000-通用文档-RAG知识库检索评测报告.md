本次按 RAG 知识库检索验收方式构造了 **10 条真实语义查询**（模拟一线工程师的典型提问场景），覆盖 20 份 UPS5000 相关的多格式文档（PDF 用户手册、PPT 培训资料、XLSX 接口图），分别评测 `keyword`、`semantic`、`hybrid` 三种检索模式。

- **hybrid 与 semantic 模式表现接近**（MRR 0.538 vs 0.544），说明当前检索以语义向量贡献为主，BM25 的补充作用未充分发挥。
- **keyword 模式整体偏弱**（MRR=0.350），中文长句和设备故障描述类查询 BM25 匹配困难。
- 本次测评将评价粒度从**文件级改为 chunk 级**，指标更严格、更真实（不再因文件命中而被判定为检索成功）。

## 2. 版本变更概要

### 2.1 评价体系升级：文件级 → chunk 级

- **原方案**：`expected_files` 只标注期望文档文件名，只要该文档的任何 chunk 出现在搜索结果中就算命中
- **问题**：假阳性——文档级命中但实际命中的 chunk 内容不相关；假阴性——相关 chunk 在其他文档中但被忽略
- **新方案**：`expected_files` 标注 `filename_chunkN` 精确到具体 chunk，匹配时检索结果中必须出现该特定 chunk 才计为命中
- **效果**：评价指标更真实地反映检索的精确度

### 2.2 Embedding 维度升级 768 → 1024

- **变更**：代码层 `Vector(768)` → `Vector(1024)`，DB 层 `ALTER TYPE` 迁移
- **原因**：1024 维度对中文语义的表达能力更强，在技术文档（中英混合）场景下能保留更多细粒度语义信息。原 768 维度对中文长句和专业术语的区分度不足。
- **影响**：所有文档需重新上传建立索引

### 2.3 中文分词支持（zhparser）

- **变更**：PostgreSQL 全文检索从 `simple` 配置切换为 `chinese`（zhparser 扩展）
- **效果**：支持中英文混合分词
- **⚠️ 待排查**：zhparser 对英文短词（型号代码如 "BF"、"FM"、"ECM"）的分词效果不如 PostgreSQL 原生的 `simple` 配置

### 2.4 切分策略重构

- **原方案**：按固定 500 字符切分，破坏段落完整性
- **新方案**：按段落（`\n\n` → `\n` → 句子）层级递进切分
- **效果**：chunk 大小差异从 6 倍缩至 1.8 倍

### 2.5 文本清洗流程

- **新增**：所有 Pipeline 在切分前调用 `_clean_text()` 统一清洗
- **清洗规则变更**：2026-07-09 删除 `_remove_repeating_footers()`。该规则将 Excel 中重复出现（15 次）的有效技术内容（"底座螺栓并联"）误判为页脚删除，违背「宁可放过不能错杀」原则

### 2.6 Excel Pipeline 重构（重写）

原版 Excel 解析存在三个严重问题：

1. **全空行被当作列名**：原版使用 `pd.read_excel` 默认行为，第一行空行导致列名变成 `0,1,2...`，解析出的文本全部是 "包含列: 0, 1, 2…" 毫无意义
2. **元数据噪声过大**：每个工作表的每列都生成一个 "列级描述" chunk，14 号 Excel 解析出 842 chunks 中 698 个是元数据
3. **未过滤空行空列**：Excel 中的格式性空行被当作有效数据，生成大量 `0: ; 1: ; 2: ; NaT` 的垃圾 chunk

**修复方案**：

- 智能检测表头行（跳过前 N 行空行，寻找首个 ≥2 非空单元格的行作为列名）
- 过滤全空行和全空列，填充空值
- 将 Excel Pipeline 从"直接生成 chunks"改为"先提取文本 → 再切分 chunks"的两阶段架构，与 Document Pipeline 统一
- 分离文本提取与切分逻辑，便于调试

**效果**：`14_机柜底座接口图.xlsx` 从 842 chunks 降至 **28 chunks**（-97%），噪声完全消除。

### 2.7 PPT 旧格式支持（全面重写）

原版 PPT 解析存在以下缺陷：

1. **不支持 .ppt 旧格式**：`python-pptx` 只支持 Office Open XML（.pptx），旧版 .ppt（二进制格式）直接报错
2. **中文路径导致 LibreOffice 转换失败**：文件路径包含中文、括号、空格时，LibreOffice headless 模式下报 `"source file could not be loaded"`，需要先复制到 `/tmp/convert_input.ppt` 简单路径
3. **解析无容错**：遇到无法读取的形状（如嵌入对象、兼容性问题）直接抛出异常终止，导致 94 页 PPT 只解析了 22 页
4. **缺少表格提取**：PPT 中的表格未处理

**修复方案**：

- .ppt → LibreOffice → .pptx → python-pptx 的双阶段转换链路
- 解析循环内加入 `try-except continue` 容错，跳过损坏形状继续处理后续页
- 提取表格内容（`shape.has_table`）
- 增加 `timeout=120s` 防止大文件转换卡死
- 重新上传后 09_培训资料.ppt 从解析失败变为 **21 chunks，14,189 字符**（原仅 2,525 字符）

### 2.8 Rerank 服务集成

- **服务**：yunwu.ai `qwen3-rerank`
- **搜索参数**：`top_k=30`, `rerank_top_k=20`
- **当前效果**：rerank 已接入但排序提升有限。从测试结果看，rerank 后的 Top5 中无关 chunk 占比仍偏高，且 `rerank_top_k=20` 的截断导致部分相关 chunk（排位 #19-#31）被挡在最终结果外。需要进一步排查 reranker 的打分分布和排序逻辑

### 2.9 清洗 bug 修复：页脚误删

- **问题**：Excel 解析后 "相连底座需用螺栓并联" 出现 15 次，被 `_remove_repeating_footers` 判定为页脚全部删除
- **修复**：删除该方法（2026-07-09）

## 4. 指标定义

| 指标 | 定义 |
| --- | --- |
| MRR | 第一个相关结果排名的倒数均值；Top-K 无相关结果记 0 |
| Recall@K | Top-K 结果中覆盖到的期望 chunks 数 / 该查询期望 chunks 总数 |
| Precision@K | Top-K 结果中属于期望 chunks 的比例 |
| NDCG@K | 归一化折损累计增益 |

**本次评估基于 chunk 级匹配**：以 `filename_chunkN` 精确匹配，不再以文档粒度判断。

- 查询中的 "如何并联" 与 Excel 表格中的 "底座螺栓并联" 措辞有差异，BM25 无法精确匹配

## 13. 改进建议

### P0：检索输入优化（查询改写与扩展）

当前系统对用户原始查询直接进行向量检索和 BM25 匹配，不做任何输入侧处理。在以下场景存在明显短板：

| 场景 | 问题 | 示例 |
| --- | --- | --- |
| **短查询** | 关键词过短，语义不明确 | "逆变器异常" → 缺乏上下文限定 |
| **口语化表达** | 与文档措辞不一致 | "指示灯不亮了" vs 文档写"指示灯异常" |
| **中英文混查** | 型号代码+中文描述组合 | "RF_Z指示灯不亮了"（RF_Z被分词器切断） |
| **同义术语** | UPS 领域存在大量同义词 | "告警"="报警"="异常"，"旁路"="Bypass" |

建议方案：

1. **同义词扩展**：构建 UPS 领域同义词词典（从现有 OP 文档资源中提取），在检索前对用户输入做同义词替换和扩展
2. **多查询改写**：一次用户查询 → 生成多个子查询（如同义改写、短语补全、中英文对照），合并多个检索结果后去重
3. **查询补全**：对短查询（≤4 个字）自动追加领域上下文，如 "逆变器异常" → "UPS 逆变器异常故障处理"
4. **HyDE（假设性文档嵌入）**：用 LLM 将用户查询改写为假设性文档片段，提升向量检索召回

### P1：记忆管理与工程师反馈机制（高优先级，长期）

当前 RAG 系统每次查询都独立走完混合检索全链路，存在以下不足：

| 问题 | 说明 |
| --- | --- |
| **无记忆状态** | 同一问题的重复查询每次都重复检索，浪费 token 和时间 |
| **无工程师经验注入** | 检索结果是纯统计/语义驱动的，无法利用工程师的历史判断 |
| **无反馈闭环** | 无法知道工程师认为哪个文档/chunk 是有用的，缺乏优化信号 |

建议设计 **上下文记忆管理模块**：

1. **检索结果反馈**：Web 端对每次检索结果提供"相关/不相关"标记，工程师可手动选中相关文档/chunk。这些反馈作为短期（会话级）和长期（知识库级）记忆存储
2. **短期记忆（会话级）**：同一次会话中，标记为"相关"的结果作为 `history` 缓存，命中相同/相似问题时直接返回，不再重走检索
3. **长期记忆（知识库级）**：积累的反馈数据 → 构建 `query → [相关文档/chunk列表]` 的映射表，作为检索的 boost 信号
4. **工程师显式经验注入**：允许工程师为常见问题手动关联文档/chunk，形成"经验知识库"，作为检索的第一优先级来源

### P2：父块上下文补充（检索时命中小块，输入 LLM 时用父块）

当前切分策略产生的 chunks 大小在 200-1000 字符之间，对于精确检索（如定位告警码）效率高，但存在以下问题：

- **小块命中但上下文不足**：检索命中了某个小块中的告警描述，但缺少该告警所属的章节、表格标题、前置条件等上下文信息
- **LLM 生成时信息不完整**：小块内容作为上下文输入 LLM 时，可能不足以让 LLM 理解完整的语境

建议方案：

1. **建立父子块索引**：切分时记录 `parent_chunk_id`，小块（子块）用于检索匹配，大块（父块）作为 LLM 输入上下文
2. **检索策略**：子块参与向量/BM25 检索排序，命中后提取父块内容作为最终上下文
3. **父块定义**：父块可以是章节段落（5000-8000 字符），子块是段落中的句子组（500-800 字符）
4. **效果预期**：检索精度（子块）和上下文完整性（父块）兼顾，目前业界 RAG 系统常用策略

### P3：修复 rerank 截断问题

当前 `rerank_top_k=20` 导致 #20 之后的期望 chunk 被截断。建议：
- 增大 `rerank_top_k` 至 50
- 或在 eval 中单独验证 reranker 打分的分布情况

### P4：表格内容 embedding 优化

Q07 的 chunk83 表格内容未进入 Top20，建议：
- 表格类 chunk 在 embedding 前做文本平坦化（flatten），去除 `[TABLE]` 标记
- 为表格 chunk 生成自然语言摘要作为检索字段

### P5：Excel 语义增强

Q09 Excel 检索完全失败，建议：
- 为每张工作表生成"工作表摘要"加入 chunk 内容
- 行列数据外包装为自然语言描述（如 "该表格记录了支脚机柜2400*850四柜的尺寸数据"）

### P6：检索重排序优化（Rerank）

当前 rerank 环节已接入但效果尚未充分发挥，主要问题：

- `rerank_top_k=20` 的截断导致部分相关 chunk 未进入 rerank 候选范围
- reranker 的打分分布未知：是否存在分数集中在某一区间导致排序区分度不足
- 未对比 rerank 开启前后的检索指标变化

建议：

1. 增大候选范围至 50，观察相关 chunk 的 rerank 得分分布
2. 对比 rerank 开启/关闭对 MRR/Recall 的影响
3. 如 reranker 打分区分度不足，考虑更换 reranker 模型或调整打分策略

## 14. 最终结论

本次 UPS5000 通用文档 RAG 知识库检索测试（chunk 级评价，10 条查询 × 20 份混合格式文档）达到 **基础可用水平**，Q01-Q06 共 6 个查询在三种模式下均有稳定召回。但 **Q07-Q10 暴露了检索系统的深层问题**：

1. **评价体系升级**：从文件级改为 chunk 级后，指标更真实、更严格，揭示了文档级评价无法发现的假阳性问题
2. **表格和 Excel 内容**是当前检索的最大短板，chunk 级评价显示这些格式的 chunk 在检索中几乎不可见
3. **rerank 截断**：20 的候选量对于覆盖低排名相关 chunk 不足
4. **语义检索在通用场景可用，在表格/告警码场景偏弱**
